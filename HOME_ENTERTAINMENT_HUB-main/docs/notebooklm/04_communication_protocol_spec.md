# 通信プロトコル仕様書

最終更新: 2026-05-09

## 1. 位置づけ

この資料は、HOME_ENTERTAINMENT_HUB における通信仕様のうち、現時点で実装済みの部分と、今後拡張予定の構想部分をまとめたものです。

- 実装済み中心: サテライト LED ノード向け UDP プロトコル
- 構想中心: 中央ノードからローカル LED ノードへ送る UART、将来の多ノード拡張

## 2. 現在の実装対象

現在の主対象は、ESP32-S3 サテライトノード 2 台に対する UDP マルチキャスト制御です。

- トランスポート: UDP Multicast
- 宛先: 239.255.0.1:6454
- 署名: 0x53
- 基本方針:
  - ACK なし
  - 必要に応じて同一パケットを複数回送信
  - ノード側で安全タイムアウトを持つ

## 3. アドレッシング

- Zone: 物理配置や系統を表す単位
- Node: 個別ノード ID
- Global: Zone 0 / Node 0 を全体一斉制御として扱う
- 注意点: Zone 1 / Node 0 は全体ではなく、誤指定としてドロップ対象になる運用がある

## 4. パケット種別

### PKT_CTRL

背景、テキストモード、明るさ、遷移、シーンなどを制御する中核パケットです。

現時点で重要な項目:

- dimmer
- bg_asset
- text_mode
- trans_ms
- flags
- execute_at_ms
- scene_id

用途:

- 背景切替
- シーン切替
- Force Reset
- 相対遅延実行
- keep-alive 相当の継続制御

### PKT_CONTENT

テキストや歌詞チャンクを送るパケットです。

現時点の主要用途:

- 歌詞の 1 行表示
- song_id / chunk_id / total_chunks / version による再構成
- 欠損時の前回行保持
- 同一 song_id で version を更新した差分更新

### PKT_NAV

ナビゲーションや単純な誘導表示向けです。

現時点では MVP 的な単色点灯や点滅中心で、空間的な左・右・前方演出は未確定です。

## 5. 実装済みの振る舞い

### 入力検証

- 署名 0x53 を確認
- CRC を確認
- Target / Global を判定

### 表示制御

- BG Asset: OFF / RAINBOW / PULSE / FIRE / OCEAN / STARFIELD
- Text Mode: OFF / LYRICS / SEAT INFO の概念を保持
- 5x8 ASCII フォントでテキストを描画
- 背景とテキストをノード内で合成

### 安全制御

- HEARTBEAT 相当の継続 CTRL が途切れると safe mode へ遷移
- Force Reset に対応
- ループ WDT を導入

### スケジューリング

- execute_at_ms = 0 は即時反映
- execute_at_ms が 1 から 60000 の場合は相対遅延として扱う
- シーン切替を将来時刻へ予約できる

## 6. シーン仕様

現時点で実装済みのシーンは次の 4 つです。

- LIVE
- HEALING
- ENTRY
- OFF

シーンは主に以下の組み合わせをまとめて切り替えます。

- 背景アニメーション
- テキスト表示モード
- 明るさ
- 遷移時間

## 7. 長時間試験で確認できたこと

30 分連続試験で、現在の UDP 制御方式が 2 台 MVP では安定することを確認済みです。

- 400ms 間隔の CTRL 継続送信
- 1 分ごとの歌詞更新
- 30 回の表示更新を欠落なく確認
- safe mode 遷移なし

これは、少なくとも現行の通信設計とノード制御ループが、MVP 条件では成立していることを示します。

## 8. まだ構想段階の仕様

以下は今後の拡張対象であり、現時点では未確定または未実装です。

- Group ID を使った論理グループ制御
- PTP 連携による高精度な未来時刻同期
- 多ノード環境での再送最適化
- 事前キャッシュと差分アセット運用
- 中央ノード完成版との統合仕様

## 9. UART 仕様の位置づけ

shared/include/hub_protocol.h には UART 最小仕様も定義されています。

- フレーム形式: STX / CMD / LEN / PAYLOAD / CRC8 / ETX
- 目的: 中央ノードとローカル LED ノードの安定接続
- 状態: 設計済みだが、現時点の主実装は UDP 側が先行

つまり、現状の通信は「UDP 実装が先に成立し、UART は仕様があるが中心経路ではない」という段階です。

## 10. NotebookLM に渡すときの読み方

- 実装済みと構想を混同しないこと
- まず UDP / 2 台 MVP の成立範囲を基準に理解すること
- TouchDesigner 側は controller 自動初期化に対応済みだが、失敗時は手動 init へフォールバックすること
- UART、Group、PTP、多ノード拡張は今後の仕様として扱うこと
