
# HOME_ENTERTAINMENT_HUB: プロジェクト・マスター仕様書 (v1.0)

## 1. プロジェクト・ビジョン
### 「日常の最高」から「国立競技場の震狂」まで
HOME_ENTERTAINMENT_HUBは、日常のリビング空間に溶け込む「最高の照明」としての顔と、国立競技場規模（100,000台）の次世代演出を両立させる、世界最大級の分散型ライティングシステムである。

- **100,000台一括制御**: 1セット約4,000 LED、合計100,000セット（約4億LED）を1ミリ秒の狂いもなく同期させる。
- **空間変革**: 日常は癒しのランタン、イベント時はスタジアム全体を巨大な一つの情報ディスプレイへと変貌させる。

---

## 2. ユースケース体系

### 2.1. 日常モード (Daily Mode)
家庭内での常設運用を前提とし、生活の質を向上させる。
1. **ライブ空間化モード**: AudioLINKCOREと同期し、映像・音楽の臨場感をリビング全体へ拡張。高速なフラッシュやカラーパルスによる没入体験。加えて、メインノードから送信される歌詞チャンクを背景レイヤーへ重畳し、歌詞情報出力装置として動作する。
2. **ヒーリングモード**: 音楽に合わせた暖色・低刺激のスロートランジションにより、一日の疲れを癒し、睡眠前の心身を整える。
3. **照明モード**: 普段使いの「最高の照明」として、単独起動で自律動作。インテリアに馴染む単色・グラデーション表示。

### 2.2. スタジアムモード (Stadium Mode)
国立競技場等の大規模施設において、10万人の来場者にパーソナライズされた体験を提供。
1. **チケットモード (Ticket Mode)**: 各ノードに座席情報、チケット詳細、会場内案内、Welcomeメッセージを個別に表示。
2. **推しアイドル・インタラクション**: 登録したアイドルの歌声や特定周波数に反応し、そのノード固有のスペシャルアニメーションを実行。専用メッセージ配信。
3. **次世代ペンライトモード**: 10万台を巨大なディスプレイとして一括制御。全ての画像・エフェクトはプリセットとして事前保存され、中央からはトリガーのみを配信。歌詞情報は一定文字数単位のチャンクで送信し、各ノードで背景とのリアルタイム合成を行う。
4. **自宅準備 & イベント後演出**: イベント前のカウントダウンや、イベント後の来場者限定スペシャルアニメーションのダウンロード。

## 3. 分散ノード・アーキテクチャ

### 3.1. 「データ分散・命令一括」の設計思想
10万台にリアルタイムで映像信号を送ることは物理的に不可能である。そのため、以下のアーキテクチャを採用する。
- **データ分散**: 重いアセット（動画フレーム、フォント、独自メッセージ）は各ノードのmicroSD/内蔵Flashに事前配置。
- **命令一括**: 中央ノードは「何番のアセットを、いつ再生するか」という数バイトのトリガーIDのみをUDPマルチキャストで送出。
- **歌詞チャンク配信**: 歌詞は曲ID・チャンクIDで管理し、リアルタイム送信時はチャンク単位で送る。急な曲順変更時は未配信チャンクのみ差分送信する。
- **自律描画**: 各ノードのESP32-S3が、受信した命令に基づきローカルデータを読み出し、リアルタイムに合成・出力する。

### 3.2. レイヤー合成 (Compositor)
ノード内部で以下の2層を独立して制御・合成する。
- **背景層 (Background Layer)**: microSD内の生RGBデータ、または数式によるプロシージャル生成。
- **テキスト層 (Text Layer)**: リアルタイム配信された歌詞、またはSD内の固有情報（座席番号等）。
- **エフェクト**: 両層に対し、独立したIn/Outエフェクト（フェード、スライド等）を適用。

## 4. 通信プロトコル詳細 (Stadium Scale Protocol)

### 4.1. セキュリティ署名 (Signature)
不正なコマンド送信や真似されたデバイスの混入を防ぐため、パケット先頭に独自の鍵を配置。
- **署名バイト**: `0x53` (Signature)。一致しないパケットは、ノード側で処理を一切行わず即座に廃棄。

### 4.2. 階層型アドレッシング
- **Global (0)**: 全ノード一斉制御。
- **Group (1-255)**: 観客席セクターやグループ単位の制御。
- **Target (1-100,000)**: 特定の個人・座席へのユニキャスト。
- **MVP運用補足**: Groupは「推しアイドル等の論理グループ」、Zoneは「会場内の物理位置」を表す別概念として扱う。1ノードは Zone 1つ + Group 0..N に所属可能とする。

### 4.3. コマンド体系
1. **Sync Preset**: アニメーション、エフェクト、パレット、明るさ、テンポ、遷移時間(16-bit)を同期。
2. **Text Update**: 表示内容の更新（Source Selectにより、リアルタイム/SD内情報/キャッシュを選択）。歌詞はチャンク送信を基本とし、`song_id`・`chunk_id`・`total_chunks`・`version`を管理して再構成する。
	- **MVP優先順位**: Realtime > Cache > SD
	- **MVPチャンク方針**: 1チャンクは 24-36文字を目安にし、上限36文字で運用する。
	- **欠損時挙動**: 欠損チャンクがある場合は直前の有効行を保持し、表示の瞬断を避ける。
3. **Force Reset**: 再生状態の強制初期化と安全停止。
	- **MVP初期化範囲**: 再生状態を初期化し、テキスト表示キャッシュを消去する。ネットワーク設定は保持する。

## 5. 10万台制御に向けた戦略

### 5.1. ネットワーク階層と時刻同期
- **階層型マルチキャスト**: メインPC → ゲートウェイ（有線） → ノード（無線）のツリー構造。
- **時刻同期 (PTP)**: 全ノードの時計をミリ秒単位で同期。パケットに「未来の再生時刻」を付与し、ネットワークの揺らぎ（ジッター）を完全に排除した同時発光を実現。
- **受け入れ基準**: MVPは同期誤差 ±20ms を許容し、最終目標は ±1ms とする。

### 5.2. 事前配信 (Pre-caching)
- **OTA更新**: イベント前にバックグラウンドでアセットを各ノードに配信。本番中のネットワーク負荷をゼロに近づける。
- **歌詞事前配信**: 標準セットリスト分の歌詞チャンクを事前配信し、イベント当日は変更が発生したチャンクのみ差分更新する。
- **MVP差分単位**: 当日運用は「曲単位」で差分配信し、必要時のみ曲内チャンク差分を使用する。

### 5.3. セキュリティ段階適用 (MVP)
- **MVP実装**: 署名バイト `0x53` と CRC チェックを必須とする。
- **将来拡張**: HMACとリプレイ防止は Phase E-1 の検討項目として段階導入する。

## 6. MVP (Phase 1) 計画
### 2台によるコアロジック検証
- **目標**: 署名チェック、レイヤー合成、2台の同時発光同期の確認。
- **構成**: ESP32-S3 x 2, WS2812B, TouchDesigner (PC)。
- **検証項目**: 署名不一致時の廃棄、背景へのテキスト重畳、Target IDによる出し分け。

### 6.1. 実機検証ステータス (2026-05-06)
- **Phase E 完了**: UDP マルチキャスト受信、署名 `0x53` / CRC 検証、Target ID 振り分け、歌詞チャンク再構成、差分更新、Force Reset を 2 台構成で確認済み。
- **Phase F 完了**: FIRE / OCEAN / STARFIELD 背景、5x8 ASCII テキスト描画、BG クロスフェード、LIVE / HEALING / ENTRY / OFF シーン切替、相対遅延実行を確認済み。
- **30分連続試験合格**: `endurance_lyrics_test.py` により 400ms 間隔の CTRL 継続送信と 1 分ごとの歌詞更新を 30 分間実施し、`M01/30 STAB` から `M30/30 STAB` まで欠落なく完走した。
- **現時点の実証範囲**: 2 台同時運用での演出同期、テキスト重畳、safe mode 未遷移の長時間運用まで確認済み。RAM 推移や Wi-Fi 再接続回数の内部観測は追加のシリアルログ試験で補完する。
