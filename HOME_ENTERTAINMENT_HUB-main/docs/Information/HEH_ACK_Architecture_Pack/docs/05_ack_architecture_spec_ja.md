## 1. 概要と共通化ビジョン

HOME_ENTERTAINMENT_HUB（HEH）プロジェクトは、「日常の最高（HOMEユース：高信頼・双方向・少数ノード）」から「国立競技場の震狂（STADIUMユース：超大規模・超低遅延・10万台）」までを同一の設計思想でカバーすることを目指しています。

通信プロトコルの断片化（フォーク）を防ぐため、本設計では**「プロトコル構造（データパケットの枠組み）は完全共通化し、ヘッダー内のフラグビットによってACKの要不要を動的に切り替える」**というアプローチ（プランA）を採用します。これにより、中央ノード（コントローラー）およびサテライトLEDノードのデータ解析コアロジックを一本化できます。

```
[ 中央ノード (コントローラー) ]
         │
         ├──( HOMEモード時 ) ──► [ パケット: FLAG_REQ_ACK=1 ] ──► [ サテライトLEDノード ]
         │                                                              │
         │◄─────────────────( UDPユニキャスト ACKパケット )──────────────┘
         │
         └──( STADIUMモード時 ) ─► [ パケット: FLAG_REQ_ACK=0 ] ─► [ サテライトLEDノード ]
                                                                        │
                                                                 (応答せず描画のみ)
```

## 2. 運用モードと通信トポロジー

### 2.1. HOME モード（日常・少台数・高機能）
* **想定環境:** 1家庭内（中央ノード1台 ↔ サテライトLEDノード 2〜16台程度）。
* **通信動作:** 中央からUDPマルチキャスト（またはブロードキャスト）で配信。パケット内の `SAT_V3_FLAG_REQ_ACK` ビットを `1`（有効）にします。
* **返信経路:** サテライトLEDノードはパケット受信時、フラグが `1` であることを確認すると、受信したUDPフレームの送信元IPアドレス（中央ノードのIP）を抽出し、**UDPユニキャスト（1対1）でACKパケット（`SAT_V3_PKT_ACK`）を返信**します。
* **UX効果:** 中央ノード（GMT020-01 TFT液晶搭載）側で、各サテライトノードの接続状態（生存・不達・CRCエラー等）をリアルタイムに把握・表示でき、日常系高級ガジェットとしての確実性とトラブルシューティング性が向上します。

### 2.2. STADIUM モード（本番・大規模・単方向）
* **想定環境:** スタジアム規模（100,000台一斉配信）。
* **通信動作:** 中央からUDPマルチキャスト配信。パケット内の `SAT_V3_FLAG_REQ_ACK` ビットを `0`（無効）にします。
* **返信経路:** **完全シャットアップ（無応答）。** サテライトLEDノードはACK要求がないため、一切の返信パケットを送信しません。これにより、無線アクセスポイントや中央ルーターの上り帯域パンク（ACKストーム）を物理的に防止します。
* **信頼性補完:** 不達対策には、ACKの代わりに「重要コマンドの冗長送信（同じパケットを3〜5回重ねて送出）」および「HEARTBEAT途絶時のノード側自律セーフモード（3秒途絶で白5%点灯など）」を併用します。

## 3. プロトコル仕様とビットマスク拡張

既存のVer 3.0プロトコル仕様（先頭署名: `0x53`）の互換性を維持しつつ、ヘッダー内の `flags` バイトおよびパケット種別（`pkt_type`）を以下のように拡張します。

### 3.1. `flags` フィールド（1バイト）のビットマスク定義
* **Bit 0 (`0x01`):** `SAT_V3_FLAG_REQ_ACK`  
    * `1` = ACK要求あり（HOMEモードのデフォルト）
    * `0` = ACK要求なし（STADIUMモードのデフォルト）
* **Bit 1 (`0x02`):** `SAT_V3_FLAG_REDUNDANT`  
    * `1` = 冗長送信された2回目以降の重複パケット（ノード側でフレームカウンタが一致すれば描画更新処理をスキップ可）
    * `0` = 初回プライマリパケット
* **Bit 2-7 (`0x04`〜`0x80`):** 将来の拡張用予約領域

### 3.2. パケット種別（`pkt_type`）の追加
ノードから中央へステータスをフィードバックするためのパケットIDを新設します。
* `0x10`: `SAT_V3_PKT_CTRL` (中核制御コマンド)
* `0x20`: `SAT_V3_PKT_CONTENT` (歌詞分割チャンクテキスト)
* `0x30`: `SAT_V3_PKT_NAV` (誘導・ナビゲーション表示用)
* `0x40`: `SAT_V3_PKT_HEARTBEAT` (同期維持・生存確認)
* `0x50`: `SAT_V3_PKT_ACK` (**新設:** サテライトノードから中央へ返す到達確認応答)
