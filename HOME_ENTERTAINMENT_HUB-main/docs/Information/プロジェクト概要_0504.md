# Project: HOME_ENTERTAINMENT_HUB - Stadium Scale Strategy

## 1. ビジョン
国立競技場規模（100,000台）のLEDノードを、1ミリ秒の狂いもなく一斉制御し、スタジアム全体を一つの巨大なディスプレイに変える。

## 2. 物理的な制約と解決策
10万台のデバイスにリアルタイムで映像信号（ユニキャスト）を送ることは、Wi-Fi帯域の限界により不可能である。そのため、以下の「分散型アーキテクチャ」を採用する。

- **制約**: 無線帯域の不足、ネットワークの揺らぎ（ジッター）、双方向通信（ACK）のパンク。
- **解決策**: 
  - **データ分散**: 重いアセットデータ（背景動画、フォント等）は各ノードのmicroSD/Flashに事前に配置する。
  - **命令一括**: 中央からは「どのIDを、いつ再生するか」という数バイトのトリガー命令のみをUDPマルチキャストで送出する。
  - **自律描画**: 各ノードが受信したIDに基づき、内部で背景とテキストをリアルタイム合成（レイヤー合成）する。

## 3. 通信プロトコル仕様 (Ver 3.0 構想)
- **トランスポート**: UDP Multicast (Art-Net 4 / sACN互換)
- **セキュリティ**: 独自署名 `0x53` (Signature) が先頭に必須。不正デバイスや誤操作を完全に排除する。
- **同期方式**: 未来のタイムスタンプを指定した「予約実行」を前提とし、PTP（時刻同期）による完璧な同時発光を目指す。
- **レイヤー構造**: 背景層（BG）とテキスト層（Text）を独立制御し、それぞれに独立した In/Out エフェクトを適用する。

## 4. MVP (Phase 1) の目標
10万台へのスケーラビリティを証明するため、まずは2台のESP32-S3で以下のコアロジックを確立する。
1. **署名チェック**: 不正パケットの破棄。
2. **レイヤー合成**: プロシージャル背景とリアルタイムテキストの合成。
3. **個別/一斉制御**: Target IDによる特定のノードへのメッセージ出し分け。

## 5. コーディング時の重要原則
- **帯域節約**: 通信パケットを1バイトでも削る設計を優先すること。
- **非同期処理**: LEDの描画（60FPS）を妨げないよう、通信解析を非同期で行うこと。
- **堅牢性**: ACKが返せない環境のため、パケットの冗長送信や自動復帰ロジックを重視すること。
