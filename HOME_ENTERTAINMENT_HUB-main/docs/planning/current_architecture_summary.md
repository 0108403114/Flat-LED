# 現在構想の整理メモ

> **最終更新**: 2026-04-xx（初版）｜**参照時点**: 2026-05-02 現在も有効  
> 最新のシステム概要は [企画案](project_proposal.md)、最新のハード選定は [BOM](../hardware/bom.md) を参照してください。

## 1. 今実現したいこと

このプロジェクトの目的は、リビング空間を音と演出で変化させる分散型ライティングシステムを作ることです。

現在の主要ユースケースは次の 2 つです。

1. ライブ映像視聴時に、音に合わせてリビング全体をライブ空間化する。
2. 就寝前に音楽に合わせて、ヒーリング空間を作る。

この目的のために、中央ノードが演出を決定し、複数の LED ノードがそれぞれの LED マトリックスを駆動する構成を目指しています。

## 2. 現在の構成

### 中央ノード

- MCU: Seeed Studio XIAO ESP32S3 Plus を第一候補
- 入力: INMP441 マイク、AudioLINKCORE 外部制御入力、ボタン
- 表示: GMT020-01 TFT
- 役割:
  - 音入力または外部制御入力を受ける
  - シーンを決める
  - プリセット情報を各 LED ノードへ送る
  - ローカル LED ノードへは UART で送る
  - サテライト LED ノードへは Wi-Fi で送る

### ローカル LED ノード

- メインノードの近くに置く LED ノード
- 中央ノードと UART 直結
- microSD からローカル演出データを読み出す
- ESP32-S3 自身で映像データを合成し、WS2812 系 LED マトリックスを駆動する

### サテライト LED ノード

- メインノードから離れた位置に置く LED ノード
- 中央ノードから Wi-Fi でプリセット情報を受ける
- 各ノードに ESP32-S3 と microSD を持つ
- 各ノードが固定識別 ID を持つ
- 各ノードがローカルで映像信号を生成して WS2812 系マトリックスを駆動する
- 初期版の識別 ID は 1 から 16 程度の短い数値を想定する
- ノードの表示名は中央ノード側の別名テーブルで管理する

### LED ノード共通方針

- LED は WS2812 系
- 4 から 6 ライン並列出力
- 1 ラインあたり約 500 LED
- 総数 2000 から 3000 LED
- ターゲット 60 FPS
- 大容量データは各 LED ノードの microSD に保持
- microSD は 4-bit SDMMC を第一候補
- 初期版は全体一斉制御と個別ノード指定制御を扱う
- 個別ノード指定演出は標準 3 秒で全体同期状態へ戻す
- FastLED は WS2812 送出層に限定して使う
- 将来用に group_id の予約領域を持つ
- group_id の 0 は全体扱いとする

## 3. 中央ノードから送る情報

現在、中央ノードから LED ノードに送るべき情報として整理できているものは次の 5 つです。

1. メインアニメーション プリセット番号
2. エフェクト プリセット番号
3. カラーパレット プリセット番号
4. テキスト情報
5. 強制リセット

考え方としては、中央ノードは大きなフレームデータを送らず、ノード側が microSD 上のローカルデータを使って描画します。
また、各サテライト LED ノードは識別 ID を持ち、全体一斉コマンドだけでなく個別指定コマンドも受けられるようにします。
初期版では、個別指定は短い数値 ID で指定し、保持時間つきの一時上書きとして扱います。
初期版の標準保持時間は 3 秒とし、ノードの表示名は中央ノード側で管理します。
また、エフェクトはアニメーションへ簡易ブレンドで重ね、テキスト表示は初期状態ではローカル LED ノードだけ有効にします。

## 4. 最初から何が変わったか

### 初期案

- Unit A と Unit B の 2 台構成を前提としていた
- Unit A と Unit B の間は SPI などの有線接続を想定していた
- LED は比較的単純な外部 LED 駆動を前提としていた
- ストレージは明確に設計へ入っていなかった
- 表示は OLED 前提で考えていた

### 現在案

- 中央ノード + 複数 LED ノードの分散構成になった
- ローカル LED ノードは UART、サテライトは Wi-Fi で接続する構成に整理された
- LED は市販マトリックスではなく、カスタム設計の WS2812 系マトリックスになった
- 各 LED ノードが microSD を持ち、ローカルアセットを読む構成になった
- 表示は OLED ではなく SPI 接続の TFT になった
- PC 再生時は AudioLINKCORE が解析済み制御信号を中央ノードへ送る構成になった
- microSD は SPI ではなく、将来性を見て 4-bit SDMMC を第一候補にした

### 変更の意味

- システムが「1 台が全部描く」設計から「各ノードが自律的に描く」設計へ変わった
- ネットワーク負荷を下げる代わりに、各ノードの責務が増えた
- スケーラビリティは上がったが、ノード設計の難度は上がった

## 5. 今の構成は最適か

結論として、現在の要求に対してはかなり筋の良い構成です。
ただし、完全に最適と断言するには、いくつか条件があります。

### 良い点

1. 中央ノードは重い映像データを配らず、制御だけに集中できる。
2. LED ノードはローカル microSD を使うので、Wi-Fi 帯域に依存しにくい。
3. WS2812 を 4 から 6 ライン並列で駆動する前提は、2000 から 3000 LED を狙う構成として妥当。
4. ローカル LED ノードだけ UART にする構成は、近距離ノードの安定性と低遅延に有利。
5. サテライト LED ノードを共通構成にできるので、量が増えたときに展開しやすい。

### 注意点

1. 60 FPS はかなりタイトで、フレーム展開、microSD 読み出し、LED 出力のバッファ設計が必須。
2. XIAO ESP32S3 Plus は有力候補だが、4-bit SDMMC と 4 から 6 本の WS2812 出力をきれいに収めるには、実際のピン割り当てを確定させる必要がある。
3. microSD を 4-bit SDMMC にする場合、ユニバーサル基板中心の試作では配線難度が上がる。
4. 強制リセットの意味を、ソフト再初期化、ブラックアウト、MCU 再起動のどれにするか決める必要がある。
5. テキスト情報の用途を、スクロール表示、演出埋め込み、デバッグ表示のどれにするか決める必要がある。

### 現時点の判断

- 要求に対する方向性は正しい。
- 大きく間違っている部分はない。
- ただし、仕様をもう少し詰めないと、後で実装コストが跳ねる箇所がある。

## 6. 仕様をもう少し詰めるべき項目

### A. 通信仕様

- UART は 1 対 1 のみで始めるか、将来的に複数ローカルノードを想定するか
- 将来のゾーン別制御でノード ID とグループ ID をどう持つか
- 強制リセット後の応答通知を返すかどうか
- Wi-Fi 側で ACK なし運用にした時の再送方針を持つかどうか

### B. アセット仕様

- アニメーションプリセット番号とアセット ID の関係
- カラーパレットプリセットの定義方法
- エフェクトプリセットがアニメーションへどう重なるか
- 生 RGB 形式の 1 フレームあたりサイズと 60 FPS 時の先読み量
- Wi-Fi 重要コマンドの複数回送信間隔
- Force Reset 完了通知を将来サテライトへ広げるかどうか

### C. LED ノード仕様

- 1 ノードあたりの正確な LED 数
- 4 ラインで始めるか、最初から 6 ラインで行くか
- レベルシフタを何系統まとめて載せるか
- 5V 電源容量の具体値

### D. MCU 選定

- XIAO ESP32S3 Plus で進めるか
- それとも LED ノードだけピン数に余裕のある ESP32-S3 ボードへ分けるか

## 7. 現時点での推奨判断

もし今ここで設計を 1 段進めるなら、次の判断を推奨します。

1. LED ノードの初期版は 4 ライン並列で始める。
2. microSD は 4-bit SDMMC を第一候補にするが、初期試作で難しければ 1-bit SDMMC へ落とせるようにしておく。
3. 通信コマンドは、アニメーション、エフェクト、パレット、テキスト、強制リセットの 5 系統で固定する。
4. 強制リセットは、まずは「出力ブラックアウト + 再生状態初期化」と定義する。
5. テキスト情報は、初期版では横スクロール表示中心、最大 32 文字程度で始める。
6. サテライト同期は同一アセット ID の一斉開始を最優先にする。
7. Wi-Fi 配信は UDP 系の軽量配信で始める。
8. フレーム列は生 RGB に近い単純形式、パターン定義は軽量演出用の補助にする。

## 8. ここまでのQ&Aで固まった初期仕様

- テキストは表示にも内部ラベルにも使う。
- テキスト表示は横スクロール中心。
- 強制リセットは全ノード一括で、ブラックアウト + 再生状態初期化。
- アセットは全ノード同一 ID 同期再生。
- プリセット数は各カテゴリ 16 程度。
- サテライトノードは自動発見。
- サテライトノードの識別は 1 から 16 程度の短い数値 ID。
- microSD 更新は初期版では手作業。
- 最優先は全ノード開始タイミング一致。
- 通信断時は最後の状態を少し維持して停止。
- 起動時は待機状態で接続待ち。
- 明るさは中央ノードから全体一括制御。
- フレーム列は生 RGB に近い単純形式。
- パターン定義はフレーム列がない時の軽量演出用。
- 将来のゾーン別演出は必要になりそうなので拡張余地を残す。
- 個別ノード指定演出は一定時間後に全体同期へ復帰。
- 個別ノード指定演出の標準保持時間は 3 秒。
- ACK は UART だけ、Wi-Fi は基本 ACK なし。
- 2026-05-17 方針更新: Wi-Fi はデフォルト ACK なしを維持しつつ、HOME 少数ノード運用でのみ `SAT_V3_FLAG_ACK_REQ` を使った動的ACKを将来拡張として許容する。
- STADIUM（100,000 SET 想定）では ACK 無効固定。信頼性は冗長送信 + HEARTBEAT + セーフモードで担保する。
- テキスト表示可否はノードごとに持つ。
- カラーパレットはアニメーション全体の色替えとして扱う。
- アニメーションプリセット番号は asset ID と 1 対 1 対応。
- Wi-Fi は重要コマンドだけ 50 ms 間隔で 3 回送る。
- group_id は将来拡張用に予約する。
- group_id の 0 は全体扱い。
- ノードの表示名は中央ノード側の別名テーブルで管理する。
- テキスト表示の初期状態はローカル LED ノードだけ有効。
- Force Reset 完了通知は UART ローカル LED ノードだけ返す。
- エフェクトは加算や乗算などの簡易ブレンドで重ねる。

## 9. 現在の総評

現在の構成は、要求に対してかなり合理的です。

- 中央ノードは制御に集中
- 各 LED ノードはローカル描画に集中
- ローカルは UART、サテライトは Wi-Fi
- 各ノードに microSD を置く
- WS2812 を 4 から 6 ライン並列で駆動

この設計は拡張しやすく、目的とも整合しています。

一方で、まだ詰めるべきなのは「通信コマンドの厳密な定義」「アセット形式」「XIAO で本当にピンが足りるか」の 3 点です。

ここが固まれば、次は実装向けの詳細設計に入れます。
