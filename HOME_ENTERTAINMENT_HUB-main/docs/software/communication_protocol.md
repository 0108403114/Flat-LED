# 通信プロトコル設計

> **最終更新**: 2026-04-xx（初版）｜**参照時点**: 2026-05-02 現在も有効  
> UARTパケット最小仕様の確定版は [実装計画](../planning/implementation_plan.md) の「UART パケット最小仕様」セクションを参照してください。

## 物理トポロジ

- 中央ノード -> ローカル LED ノード: UART
- 中央ノード -> サテライト LED ノード群: Wi-Fi
- ローカル LED ノードとサテライト LED ノードは、同じ論理コマンド構造を共有する。

## 中央ノードから送る情報

1. メインアニメーション プリセット番号
2. エフェクト プリセット番号
3. カラーパレット プリセット番号
4. テキスト情報
5. 強制リセット

## 初期版で固まった前提

- テキストは表示と内部ラベルの両方に使う。
- テキスト表示は横スクロール中心にする。
- 強制リセットは全ノード一括で送り、意味は「即ブラックアウト + 再生状態初期化」とする。
- アセットは全ノードが同じアセット ID を再生する前提にする。
- 各カテゴリのプリセット数は初期版で 16 個程度を想定する。
- サテライトノードは固定 ID を持ち、起動時に自動発見する。
- microSD 上のアセット更新は初期版では手作業で行う。
- 同期で最優先するのは全ノードの開始タイミング一致とする。
- Wi-Fi 断、UART 断のどちらでも最後の状態を短時間維持してから停止する。
- ノード起動直後は待機状態で接続待ちにする。
- 明るさ制御は中央ノードから全体一括制御する。
- フレーム列データは初期版では生 RGB に近い単純形式を優先する。
- パターン定義データはフレーム列がない時の軽量演出用として使う。
- 将来のゾーン別演出は必要になる見込みなので、ノード ID やグループ拡張余地を残す。
- 初期版のサテライト識別子は 1 から 16 程度の短い数値 ID にする。
- 初期版の制御粒度は全体一斉と個別ノード指定にする。
- 個別ノード指定演出は、標準 3 秒で全体同期へ復帰する。
- コマンド受信確認は UART のみ ACK を返し、Wi-Fi は基本 ACK なしで運用する。
- テキスト表示可否はノードごとに持てるようにする。
- カラーパレットプリセットはアニメーション全体の色替えとして使う。
- 各 LED ノードは microSD アセットと受信コマンドを合成し、最終フレームを自分で生成する。
- FastLED は WS2812 送出層だけに使う。
- アニメーションプリセット番号は初期版では asset ID と 1 対 1 対応にする。
- 将来のグループ制御に備えて group_id の予約領域を持つ。
- ノードの表示名は中央ノード側の別名テーブルで管理する。
- テキスト表示の初期状態はローカル LED ノードだけ有効にする。
- group_id の初期値 0 は未使用ではなく全体扱いとする。
- エフェクトは加算や乗算などの簡易ブレンドでアニメーションへ重ねる。

## 推奨コマンド種別

### Sync Preset

- 目的: 再生中アニメーションと演出補助情報を同期する。
- 主な項目:
  - scene_id
  - target_node_id
  - group_id
  - mode_id
  - animation_preset_id
  - effect_preset_id
  - palette_preset_id
  - hold_ms
  - brightness
  - tempo_bpm
  - fade_ms

### Text Update

- 目的: LED ノード側で文字列を表示または演出へ反映する。
- 主な項目:
  - scene_id
  - text_length
  - text
- 初期版の最大文字数は 32 文字程度を想定する。

### Force Reset

- 目的: 再生状態を強制初期化し、ブラックアウトまたは再起動へ遷移させる。
- 主な項目:
  - scene_id
  - flags
- 初期版では全ノード一括だけを対象にする。

## 通信方針

- UART と Wi-Fi でペイロード構造をそろえる。
- Wi-Fi 側は大容量フレームを送らず、プリセット番号と同期パラメータだけを送る。
- 各 LED ノードは受信したプリセット番号をキーに、microSD 上のローカルデータを読む。
- 初期版では UDP 系の軽量配信を優先する。
- UART は ACK を返し、中央ノードが送達確認できるようにする。
- Wi-Fi は一斉配信を優先し、初期版では ACK なしで運用する。
- Wi-Fi は scene 切替や Force Reset などの重要コマンドだけ 50 ms 間隔で 3 回送る。

## ACK アーキテクチャ方針（2026-05-17 反映）

### 目的

- HOME の少数ノード運用では到達可視化を強化する。
- STADIUM の超大規模運用では ACK ストームを回避する。
- 同一プロトコル枠内で運用モードを切り替え、実装分岐を最小化する。

### 採用方針

- ローカル UART: ACK を使用する（従来方針を維持）。
- Wi-Fi HOME 運用（少数ノード）: `SAT_V3_FLAG_ACK_REQ` を有効化した場合のみ ACK を許可する。
- Wi-Fi STADIUM 運用（100,000 SET 想定）: ACK は無効固定とし、冗長送信 + HEARTBEAT + セーフモードで担保する。

### 互換性ルール（重要）

- 既存 `shared/include/hub_protocol.h` のフラグ割当は維持する。
  - `bit0 = PTP_VALID`
  - `bit1 = UNICAST`
  - `bit2 = ACK_REQ`
  - `bit3 = REDUNDANT`
- 互換性維持のため、ACK_REQ を bit0 に再配置する変更は採用しない。

### 実装メモ

- ACK 応答パケット種別（例: `0x50`）の追加は将来拡張として検討する。
- ただし STADIUM 向け運用プロファイルでは ACK 返信処理を無効化する。
- HOME 向け ACK 集計は非ブロッキングで行い、受信欠落時も描画ループを止めない。

### 最終決定（2026-05-17）

- 現行標準運用は「UART ACK あり / Wi-Fi ACK なし」を正式採用する。
- HOME 向け動的ACKは「将来拡張（検証プロファイル）」として採用し、常時有効化はしない。
- STADIUM は ACK 無効固定とする。

### HOME 動的ACKの有効化条件

以下を満たしたときのみ、HOMEプロファイルで ACK_REQ を有効化してよい。

1. ノード数が少数（目安: 16台以下）である。
2. ACK 集計が非ブロッキングで、描画ループを阻害しない。
3. ACK 欠落時に演出停止せず、冗長送信と HEARTBEAT で継続可能。
4. STADIUM プロファイルでは同一ビルド内でも ACK 経路を無効化できる。

## 実装メモ

- UART はローカル LED ノード向けの低遅延・安定経路として使う。
- Wi-Fi はサテライト向けの一斉配信経路として使う。
- 強制リセットはソフトリセットだけでなく、出力ブラックアウト命令として扱えるようにする。
- Force Reset 完了通知は初期版では UART 接続のローカル LED ノードだけ返す。
